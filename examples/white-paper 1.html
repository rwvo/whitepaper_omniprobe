Executive Summary

Distill is a working framework for evaluating what a language model knows about a defined subject domain by grounding the evaluation in a curated source corpus, converting that corpus into a structured map of the domain, distilling evidence-backed claims from that map, probing models against those claims, and adjudicating the resulting answers against retrieved evidence. The implementation in this repository is designed for questions that conventional benchmarks handle poorly: not whether a model can pass a generic exam, but whether it demonstrably possesses reliable knowledge of a specific field such as chip design, compiler optimization, or another bounded technical discipline.

The system is deliberately claim-first rather than question-first. Ad hoc question banks are difficult to normalize, difficult to deduplicate, and difficult to verify consistently. Distill instead treats the domain as a set of atomic, testable, source-linked claims. Questions and prompts are derived from those claims later. This provides a more defensible evaluation reference because the core unit is not a prompt phrasing but a proposition that can be traced back to evidence, assigned to a domain concept, audited for quality, weighted by importance, and revisited as the corpus evolves.

Distill also addresses a practical problem that many model-evaluation efforts ignore: the need for transparent operational artifacts. The repository contains a concrete end-to-end pipeline for domain-pack generation, automated source discovery, ingestion into an evidence bundle, atlas construction, heuristic and LLM-assisted claim extraction, claim auditing and promotion into a canonical claim bank, probe generation, live model evaluation through OpenAI-compatible Responses APIs, evidence-grounded judged evaluation, and final consultant-style reporting in HTML, PDF, and PowerPoint formats.

That does not make the system objectively correct in an absolute sense. It does make it auditable. Distill turns a vague question such as “What does model X know about chip design?” into a chain of explicit assumptions, artifacts, transformations, and judgments that can be inspected, criticized, rerun, and improved.

The Problem Distill Solves

Evaluating subject-matter knowledge in language models is harder than it first appears. A model may produce fluent, convincing language while still being shallow, inaccurate, or unstable. A benchmark built from a small hand-authored question set can reveal some weaknesses, but it rarely provides a defensible answer to the broader question of domain mastery. Those efforts are usually narrow, unevenly distributed across subdomains, informally graded, and poorly traceable back to source evidence.

Distill attacks those problems by treating knowledge evaluation as a corpus-and-claims problem rather than only a prompting problem. The system begins with a structured definition of the domain, builds an evidence base, derives an atlas that reflects the conceptual terrain of that evidence, converts the atlas and evidence into a canonical claim bank, and then uses that claim bank as the substrate for probing and adjudication.

The practical question Distill tries to answer is not “Can the model answer a handful of questions?” but “Does the model reliably support, explain, and apply a defensible body of evidence-backed domain claims?”

The Central Design Thesis: Claim-First, Source-Grounded Evaluation

The most important conceptual decision in Distill is that claims, not prompts, are the reference unit of knowledge. A claim is an atomic technical statement such as a definition, mechanism, constraint, tradeoff, failure mode, or misconception. Claims can be linked directly to evidence chunks, mapped to atlas nodes, deduplicated across sources, assigned importance and difficulty, and converted into multiple prompt families without changing the underlying proposition being tested.

This changes the evaluation problem materially. Distill does not try to hand-author a “complete set of questions.” Instead, it tries to build a defensible claim bank that approximates the important knowledge content of the subject domain. Prompt generation is then a downstream rendering step. This makes comprehensiveness more tractable because the system can ask whether the claim bank covers the atlas, whether the atlas covers the declared scope, and whether the corpus supports the atlas.

System Architecture Overview

At a high level, Distill transforms a subject definition into a judged knowledge profile through a sequence of reviewable artifacts. The architecture is intentionally artifact-centric. Every major stage emits machine-readable JSON and a corresponding HTML report, allowing operators to inspect intermediate outputs rather than trusting a monolithic black box.

Corpus Discovery and Source Filtering

Any claim that Distill later treats as evaluative truth originates, directly or indirectly, in the corpus. The implemented source-discovery flow begins with the domain pack’s seed queries, seed sources, and required source types. Distill can then expand the query set using a configured model through an OpenAI-compatible Responses API. The expanded queries are tagged with subdomain, preferred source type, and priority. If a coverage plan is supplied, query expansion can use coverage gaps and existing-domain saturation as part of the expansion context. If model-assisted expansion fails, the system falls back to deterministic query construction rather than stopping the pipeline.

Distill then harvests public search results from multiple search backends, normalizes and deduplicates URLs, unwraps redirect links, fetches page metadata where possible, and applies heuristic ranking. The ranking logic considers lexical overlap with the domain, source-type hints, trust-tier hints, query provenance, backend corroboration, and domain quality heuristics. The implementation also contains explicit blocked-domain logic and off-topic text filters to reject irrelevant material.

1. Scope Steering

The domain pack narrows the subject boundary before any search or ranking logic runs.

2. Query Expansion

LLM-assisted and fallback query generation broaden search coverage while preserving subdomain intent.

3. URL Harvesting

Multiple public search backends are used to gather candidate URLs and snippets.

4. Metadata and Ranking

Metadata harvesting, deduplication, trust-tier hints, and heuristic scoring reduce noisy or off-domain material.

5. LLM Selection

A structured source-selection pass chooses candidates with deterministic fallbacks and caps.

6. Merge and Review

Selected results are merged with seed and prior manifest entries into a reviewable source manifest.

This multi-stage design is central to Distill’s claim of defensibility. The system does not assume that a model can hallucinate a good bibliography and be trusted. It combines public discovery, metadata, heuristics, constraints, and human-reviewable artifacts to reduce that risk.

Ingestion and the Evidence Bundle

Source discovery identifies what to read. Ingestion determines what later stages are actually allowed to reason over. The implemented ingestion stage fetches web content where possible, reads local files when locators point to filesystem content, normalizes HTML and Markdown into plain text, preserves source metadata, and chunks documents into overlapping text windows. The current chunking strategy is intentionally small enough for retrieval and claim extraction, but large enough to preserve local context.

Each ingested document receives a stable document_id, and each chunk receives a stable chunk_id. Document records preserve title, locator, resolved locator, source type, trust tier, sensitivity, word count, content hash, ingest status, and operator notes. Chunk records preserve source lineage, sensitivity, ordering, word offsets, and chunk text.

Document Record

Title, locator, resolved locator, source type, trust tier, sensitivity, ingest status, content hash, and notes.

Chunk Record

Stable chunk identifier, source and document lineage, sensitivity, ordering, offsets, and normalized text.

Why It Matters

Later judged evaluations can cite exact chunks rather than vaguely citing sources at document level.

Building the Domain Atlas

Distill’s atlas is the bridge between a raw corpus and a usable evaluation reference. Without some structured representation of the subject, a large corpus remains too diffuse to measure coverage, generate balanced claims, or explain results by subdomain. The atlas-building implementation starts from the in_scope subdomains declared in the domain pack. Those subdomains form explicit top-level lanes.

For each subdomain, Distill identifies relevant evidence chunks through lexical similarity and uses those chunks to scaffold the atlas. It creates lane nodes for fundamentals, tradeoffs, failure modes, and misconceptions, then extracts phrase candidates from relevant sentences to propose additional nodes. These proposed nodes are typed heuristically based on technical markers. The result is a corpus-backed atlas rather than an ontology imagined only by an LLM.

The atlas can then be refined with an LLM-assisted pass that is constrained to allowed evidence chunks. A subsequent normalization pass merges duplicate or overlapping nodes, rewrites edges, induces additional structural links, and records merge reviews and redirect maps. The atlas is therefore a hybrid artifact: seeded by the domain pack, expanded from evidence, cleaned with LLM assistance, and normalized deterministically.

From Atlas to Claims

If the atlas is the map of the domain, the claim bank is the operational statement of what knowledge within that map should be testable. Distill currently supports two complementary claim-generation paths. The heuristic path extracts candidate sentences from evidence chunks, classifies them into claim types such as definition, mechanism, procedure, tradeoff, or failure mode, estimates difficulty and importance, and links them to the most relevant atlas nodes.

The LLM-assisted path asks a configured extractor model to pull atomic technical claims from a source chunk using only the provided excerpt and a bounded list of atlas topics. The extractor is constrained by a schema that requires claim text, claim type, difficulty, importance, and an atlas hint. Heuristic extraction is cheap, deterministic, and recall-oriented. LLM-assisted extraction is better at paraphrasing dense source text into cleaner canonical statements and surfacing nuanced claims that sentence heuristics might miss.

Claim Quality Scrubbing and Promotion

Claim extraction must over-generate, but evaluation cannot tolerate semantic garbage. Distill therefore includes an explicit claim-quality subsystem rather than assuming every extracted sentence is a valid evaluative claim. The deterministic quality layer rejects placeholders, malformed fragments, prompt echoes, schema leakage, conversational filler, meta-discourse, and heading-like text. This is the mechanism that filters out junk such as "string" or prompt residue like Return only a JSON object with exact shape..., both of which are common failure modes when LLMs are used inside extraction pipelines.

After deterministic screening, Distill can run an LLM-assisted audit pass. The auditor receives the cleaned candidate text, source excerpt, linked atlas labels, and heuristic scores. It must return a structured verdict of accept, revise, or reject, together with quality scores, failure reasons, a suggested rewrite when appropriate, and rationale. Distill then reconciles heuristic and model scores, applies additional rewrite-salvage logic, blocks rewrites for off-domain or unsupported claims, and only promotes revisions that remain source-grounded and semantically coherent.

Promotion into the canonical claim bank is intentionally conservative. Candidates can be filtered out for low importance, lack of atlas linkage, weak evidence score, weak document support, or semantic noise. Duplicates are merged, evidence references are unioned, corroboration is retained, and the highest sensitivity label is preserved.

Coverage Analysis and Iterative Expansion

A claim bank is only useful if it meaningfully covers the atlas. Distill therefore includes a coverage-analysis stage that evaluates how well atlas nodes are supported by claims, probe families, and available evidence. The coverage planner computes per-node target claim counts and target probe counts based on node type and importance. It measures aligned claims, claim-type diversity, available evidence, and probe availability to produce node-level coverage scores and statuses such as covered, partial, or gap.

This is not a cosmetic reporting step. Coverage analysis is what allows Distill to perform iterative, coverage-driven corpus expansion. When coverage is weak in a lane such as failure modes or tradeoffs for a given subdomain, that signal can be fed back into source discovery so that query expansion targets those under-supported areas.

Probe Design and What Models Are Asked to Do

Once the canonical claim bank exists, Distill turns claims into executable prompts. The current implementation emits three probe families: claim_recall, claim_explanation, and claim_application. These probe families matter because knowledge is not one-dimensional. A model may be able to restate a claim but fail to explain it mechanistically. It may explain a concept but fail to apply it under realistic constraints.

All live model interactions are routed through an OpenAI-compatible Responses API abstraction. Providers are defined in distill.toml with configurable base_url, API keys, optional CA bundle environment names, static headers, and environment-backed headers. Model aliases are separate from providers, which means the same logical evaluation pipeline can route traffic to OpenAI, OpenRouter, AMD Gateway, Ollama, or another compatible endpoint by changing configuration rather than rewriting evaluation code.

The probe prompt asks the model to return a JSON object with a standardized shape containing a judgment, answer, key points, and confidence. Distill then parses this structure, extracts candidate response claims from the answer and key points, and performs a preliminary heuristic verification step before the stronger judged-evaluation pass.

How Correctness Is Actually Evaluated

Distill’s correctness machinery is best understood as a two-stage process. The first stage happens at probe time. After the model returns a structured response, Distill extracts candidate response claims from the answer and key points. It then compares those claims to the canonical claim and its evidence references using lexical and structural heuristics. This yields an initial probe verdict such as supported, partially_supported, needs_review, contradicted, or unsupported.

The second stage is judged evaluation, which is the stronger and more defensible correctness pass. For each extracted response claim, Distill retrieves the top evidence chunks from the retrieval index, constructs a bounded judge prompt that includes the canonical claim, the extracted response claim, the allowed chunk IDs, and the evidence text, and asks the configured judge model to use only that evidence. The judge must return a structured verdict from the set supported, partially_supported, contradicted, unsupported, or ambiguous, plus a short rationale and the evidence chunk IDs used.

The implementation’s main safeguard is that the judge is asked to decide within a constrained evidence envelope rather than from unconstrained world knowledge.

The Role of the Judge Model

The judge model is one of Distill’s most important explicit assumptions. The system treats the configured judge as the authoritative adjudicator for evidence-grounded correctness classification. That does not mean the judge is presumed infallible. It means the system operationally delegates the final interpretation of evidence-model agreement to the judge model because a fixed set of deterministic rules is not expressive enough to grade varied, paraphrastic technical answers at scale.

In practical terms, the judge serves three functions. It helps generate and critique domain packs, it can assist with source selection and explanatory report narrative, and most importantly it adjudicates model claims against retrieved evidence. The credibility of final results therefore depends on accepting that the configured judge is sufficiently authoritative, sufficiently disciplined under constraint, and less error-prone in this adjudication role than the evaluation models being assessed.

Privacy Labels and Sensitivity Propagation

Distill includes a multi-level sensitivity model so that the same knowledge-evaluation pipeline can also surface possible restricted-knowledge exposure. The supported sensitivity levels are public, private, and secret. These labels can be attached to source manifest entries together with an optional sensitivity_reason.

The sensitivity label is normalized at source ingestion time and then propagated through the entire pipeline. Documents carry sensitivity. Evidence chunks inherit it. Claim candidates inherit it from supporting chunks. Canonical claim-bank entries preserve or elevate it during duplicate merging. Probes inherit claim sensitivity and are tagged with a disclosure mode. Judged evaluation results retain sensitivity so that support statistics can be broken out by public, private, and secret content.

The correct interpretation of these labels is important. A model that correctly answers a secret claim does not automatically prove training-data leakage. The answer may have been inferred from public facts, approximated from adjacent knowledge, or derived from overlap between restricted and public corpora. Sensitivity is therefore an exposure signal, not standalone proof of compromise.

Why the Claim Bank Can Function as an Objective Reference

No knowledge benchmark can be perfectly objective in a philosophical sense. Distill instead aims for operational objectivity: a reference that is explicit, inspectable, source-backed, and reproducible enough that informed reviewers can understand why results came out the way they did. The claim bank supports that goal because it is derived from multiple sources rather than a single authority, linked to an atlas, evidence-backed at the chunk level, scrubbed for semantic noise, promoted conservatively with importance and evidence thresholds, and annotated with corroboration, support-document counts, and sensitivity provenance.

This is a stronger position than a benchmark that simply publishes a set of prompts and answers. In Distill, the benchmark reference is a corpus-backed set of technical claims whose provenance and structural relationship to the subject are visible.

Reporting and Executive Communication

The repository does not stop at scoring. It includes a reporting system that turns judged runs, coverage data, and corpus metadata into professional presentation outputs. The implemented reporting layer assembles an integrated bundle that includes model-by-model judged support rates, subdomain views, sensitivity breakdowns, representative examples, source references, and consultant-style narrative interpretation.

When a configured authority model is available, Distill can also use that model to generate explanatory prose for sections such as executive overview, methodology, chart interpretation, key findings, and glossary terms. The output can be rendered as HTML, converted to PDF through a headless Chrome path when available, and summarized into an editable PowerPoint deck.

Operational Characteristics: Configuration, Concurrency, Checkpointing, and Resume

Distill is built to operate across both local and hosted model providers. All model calls go through a single OpenAI-compatible Responses API client with configurable providers and model aliases. This provides a clean separation between the logical role of a model and the transport endpoint used to reach it.

The implementation includes bounded concurrency for the major LLM-bound stages, including atlas refinement, LLM claim extraction, claim scrubbing, evaluation, and judged evaluation. The concurrency layer preserves deterministic output ordering even when requests finish out of order. Long-running evaluation and judge operations also support live checkpointing and resume. Progress is written incrementally rather than only at the end of a run, with configurable checkpoint cadence and heartbeat-based progress refreshes.

From an operational perspective, these features matter as much as the conceptual design. A domain-specific evaluation may require thousands of remote model calls. Without concurrency, observability, and checkpointing, such runs are brittle and uneconomical.

Assumptions, Limitations, and Acceptance Conditions

To accept Distill’s results, a reviewer must accept several explicit assumptions. These are not incidental footnotes. They are the conditions under which the evaluation should be interpreted.

Judge Authority

The final correctness layer depends on the configured judge model being sufficiently authoritative and disciplined under constrained evidence prompts.

Corpus Quality

Distill can only evaluate what is represented in the corpus and transformed into claims. Shallow or biased corpora bound the quality of results.

Claim Coverage Proxy

Broad, well-structured claim coverage is treated as a practical proxy for broad, well-structured domain knowledge coverage.

Closed-Book Semantics

The headline interpretation is about native model knowledge without retrieval or tool assistance unless those capabilities are evaluated separately.

Sensitivity Interpretation

Performance on restricted claims is a signal for further investigation, not standalone proof of training-data leakage.

These assumptions define the acceptance boundary of the system. A reviewer who rejects them may still find Distill useful as an exploratory tool, but should be cautious about treating its outputs as definitive.

Why Distill Is Well-Aligned to the Model-Knowledge Evaluation Problem

The architecture implemented in this repository is well aligned to the actual problem of evaluating model knowledge because it decomposes the problem into the right units. The domain pack defines what should count as the domain. The corpus discovery process defines what evidence should be considered. The atlas defines what conceptual terrain should be covered. The claim bank defines what atomic propositions should be testable. The probe bundle defines how those propositions should be exercised. The judged evaluation defines how responses should be checked against evidence. The report defines how the resulting knowledge profile should be communicated.

That decomposition is more than a software organization choice. It is an epistemic discipline. It forces the system to state what subject it means, what evidence it trusts, what propositions it thinks matter, what kinds of knowledge it tests, and what adjudicator it empowers to decide correctness.

Conclusion

Distill is not merely a prototype for benchmarking language models. It is a functioning framework for building evidence-grounded, domain-specific knowledge assessments. Its most important contribution is not a single algorithm but a disciplined pipeline: define the domain, curate the corpus, structure the ontology, distill atomic claims, scrub and promote them conservatively, probe models through a uniform interface, adjudicate answers against evidence, and communicate the results in a form that decision-makers can understand.

The system’s outputs are only as credible as its corpus, atlas, claim bank, retrieval quality, and judge model. Distill does not pretend otherwise. Instead, it makes those dependencies explicit and reviewable. For organizations trying to understand what an open-weight or hosted model actually knows about a sensitive or strategically important domain, that explicitness is the difference between a demo and an evaluative instrument.
