Omniprobe

Source-level GPU memory access analysis for HIP and Triton kernels. Omniprobe fills the gap between hardware-counter profilers and the per-instruction, per-source-location insight developers need to diagnose uncoalesced accesses, LDS bank conflicts, and memory inefficiencies in AMD GPU workloads.

Executive Summary

Modern GPU applications are increasingly limited not by compute throughput but by memory access efficiency. An AMD CDNA GPU can deliver teraflops of peak compute, yet a single kernel with uncoalesced global memory accesses or LDS bank conflicts can leave the majority of that bandwidth unutilized. Existing profiling tools report hardware performance counters and roofline metrics, but they do not tell the developer which source line caused the inefficiency or which specific lanes in a wavefront accessed conflicting memory banks.

Omniprobe bridges this gap. It instruments GPU kernels at compile time using LLVM IR passes, intercepts kernel dispatches transparently at runtime via the HSA API, streams per-instruction memory access data from device to host in real time, and reports findings grouped by source file, line, and column. The developer sees exactly which memory operations are inefficient, what the hardware actually did, and what the fix should look like.

Omniprobe supports both HIP and Triton frontends, works with production libraries such as rocBLAS and hipBLASLt, and requires no modifications to application source code. Instrumentation creates isolated kernel clones, so non-instrumented execution paths carry zero runtime overhead.

Omniprobe does not replace hardware-counter profilers. It complements them by providing the source-level memory access detail they were not designed to capture.

The Analysis Gap

AMD GPU developers today have access to powerful profiling tools. rocprofv3 provides kernel-level hardware counter data and traces. Omniperf offers roofline analysis, occupancy metrics, and memory hierarchy statistics. Omnitrace captures timeline traces for multi-threaded CPU+GPU workflows. These tools answer critical questions about overall kernel performance, resource utilization, and system-level bottlenecks.

But there is a class of performance problems these tools cannot diagnose precisely:

Uncoalesced Accesses

When 64 lanes in a wavefront access scattered global memory addresses, the hardware issues multiple cache line fetches instead of one. Counter-based tools report high L2 traffic, but cannot pinpoint which load instruction is the culprit or which lanes diverge.

LDS Bank Conflicts

When multiple lanes access the same LDS bank simultaneously, accesses serialize. Hardware counters report aggregate LDS stalls, but cannot show which shared memory access pattern causes the conflict or for which specific wavefront shape.

Source-Level Attribution

Counter-based tools operate at the kernel level or, at best, the ISA instruction level. Mapping a problematic ISA instruction back to the exact source line, especially after compiler optimizations, requires manual effort and ISA expertise.

Omniprobe fills this gap by operating at the per-instruction, per-wavefront granularity and mapping every finding back to a source file and line number via DWARF debug information. The developer does not need to read ISA or interpret counter ratios. The output directly says: "line 42 of kernel.hip: 64 lanes touched 16 cache lines instead of the minimum 2 — 8x memory bandwidth waste."

Architectural Invariants

Zero-overhead design
Original kernels are never modified. Instrumentation creates clones with an _inst suffix. Non-instrumented execution paths carry no runtime overhead whatsoever.
Transparent interception
The application under analysis requires no source code changes. The interceptor hooks HSA API calls and swaps kernels at the dispatch level.
Non-blocking analysis
Message handlers run on separate host threads and never block GPU dispatch. The GPU does not wait for the host to process messages.
ISA-level accuracy
The kernel database correlates IR-level DWARF debug info with actual ISA instructions, correcting for compiler optimizations that change access sizes (e.g., a 4-byte IR load compiled to a 16-byte buffer_load_dwordx4).

Compile-Time Instrumentation

Omniprobe's instrumentation subsystem consists of LLVM IR pass plugins that are loaded by the compiler during kernel compilation. Each plugin implements the clone-and-instrument pattern: it creates a copy of every kernel function, appends _inst to the clone's name, adds a void* dh_comms_descriptor as the final argument, and inserts instrumentation calls at selected instructions.

Available Plugins

Address Messages

Instruments load and store instructions. Inserts v_submit_address() calls that capture the 64-lane address vector and DWARF source location. Used by the Memory Analysis, Address Logger, and Heatmap analyzers.

Basic Block Start

Instruments basic block entries. Inserts s_submit_wave_header() calls that record wavefront entry into each basic block. Used by the Basic Block Logger and Analysis handlers.

Basic Block Interval

Instruments basic block timing by inserting s_submit_time_interval() calls at basic block boundaries. Captures execution time per block for profiling.

Scope Control

Instrumentation can be restricted to specific source files and line ranges at compile time using the INSTRUMENTATION_SCOPE environment variable. The syntax file.hip:10:50 limits instrumentation to lines 10–50 of that file. Multiple ranges can be combined with commas and semicolons. This is particularly valuable when analyzing large kernels where only a specific code region is of interest, as it reduces message volume and analysis overhead.

Dual LLVM Variant Compilation

Because instrumentation plugins are loaded directly into the compiler, they must match the LLVM version of the host compiler. ROCm and Triton use different LLVM builds, so each plugin is compiled twice: once against the ROCm LLVM and once against the Triton LLVM. The build system handles this automatically.

Runtime Dispatch Interception

At runtime, the Omniprobe interceptor operates as an HSA tools library registered through rocprofiler-sdk. It hooks three critical HSA API functions to achieve transparent kernel swapping without any application modifications.

hsa_queue_create
Registers queues and HSA agents so the interceptor can track which GPU will execute each dispatch.
hsa_executable_symbol_get_info
Captures kernel object handles into an internal map and registers them in the kernel cache. This is where Omniprobe discovers which kernels are available and whether instrumented variants exist.
OnSubmitPackets
Intercepts dispatch packets before they reach the GPU. The doPackets() function decides whether to swap in the instrumented kernel variant. If swapped, fixupKernArgs() appends the dh_comms_descriptor pointer as the final kernel argument.

On-Demand Lazy Scanning

Production GPU libraries like rocBLAS contain over 12,000 kernel symbols. Eagerly scanning all code objects at startup caused delays exceeding 10 minutes. Omniprobe's solution is on-demand scanning: at dispatch time, if the kernel is not yet in the kernel database, scanCodeObject() is called for that specific code object. Startup is near-instantaneous regardless of library size.

The on-demand design means Omniprobe adds negligible startup latency even when analyzing applications that link against dozens of GPU libraries with thousands of kernel symbols.

Library Filtering

The interceptor includes a configurable library filter that controls which GPU libraries are scanned and intercepted. Libraries can be included or excluded by path using a JSON configuration file. This allows developers to focus analysis on specific libraries (e.g., only hipBLASLt) while ignoring others.

Device-Host Communication

The dh_comms subsystem provides the shared-buffer infrastructure that enables instrumented kernels to stream analysis data to the host during execution. Unlike post-mortem profiling approaches, data flows from GPU to host in real time while the kernel is still running.

dh_comms Descriptor

A struct passed to the instrumented kernel as its extra argument. Contains pointers to shared sub-buffers that the kernel writes to and the host reads from.

Message Format

Each message_t contains an array of 64 addresses (one per wavefront lane), an address-space identifier (FLAT, GLOBAL, SHARED, CONSTANT), and DWARF source location information (file, line, column).

Pool Management

The comms_mgr pools dh_comms objects with checkout/checkin semantics. One pool per HSA agent. Pools grow on demand. Thread-safe access via mutex.

Handler plugins are attached to dh_comms objects at checkout time. When the kernel completes and the object is checked back in, the polling thread stops, handlers produce their reports, and the object is cleaned up. This per-dispatch lifecycle ensures that handler state is isolated between kernel invocations.

Memory Analysis Engine

The memory analysis handler is the flagship analysis component. It processes the stream of 64-lane address messages and detects two categories of performance problems.

Uncoalesced Global Accesses

For each global memory instruction, the handler counts the distinct cache lines touched by all 64 lanes in the wavefront. It compares this against the theoretical minimum (the number of cache lines needed if all lanes accessed contiguous addresses). The ratio indicates memory bandwidth efficiency: a ratio of 1.0 is perfectly coalesced, while 16.0 means 16x bandwidth waste.

LDS Bank Conflicts

For shared memory (LDS) accesses, the handler partitions the 64 lanes into conflict sets based on bank mapping and access size, then detects when multiple lanes within a set target the same bank. Bank conflicts cause serialized access, degrading LDS throughput.

Architecture-Aware Cache Line Sizes

Cache line analysis is parameterized by GPU architecture. The handler uses the correct L2 cache line size for the target GPU:

gfx906 / gfx908: 64-byte lines
gfx90a / gfx940 / gfx941 / gfx942: 128-byte lines

ISA-Level Correction

A critical accuracy feature. The compiler may optimize IR-level memory accesses: a 4-byte load in source code might become a 16-byte buffer_load_dwordx4 in the generated ISA. The memory analysis handler uses the kernel database to correlate IR-level DWARF information with actual ISA instructions, ensuring that cache line and bank conflict calculations use the true hardware access size rather than the IR-level approximation.

Without ISA-level correction, analysis results could be misleading. A seemingly uncoalesced 4-byte access might actually be a perfectly coalesced 16-byte vectorized load. Omniprobe's kernel database integration eliminates this class of false positives.

Pluggable Handler Architecture

The memory analysis handler is one of several analysis plugins. Handlers are loaded via dlopen() with a C factory interface (getMessageHandlers()). New analyzers can be added as shared libraries without modifying the Omniprobe core. Built-in handlers include:

Memory Analysis
Detects uncoalesced accesses and LDS bank conflicts. Reports efficiency ratios per source location.
Message Logger
Raw trace output of all address messages. Useful for custom post-processing.
Basic Block Analysis
Counts basic block entries per wavefront. Useful for control-flow divergence analysis.

HIP & Triton Workflows

Omniprobe supports two GPU programming frontends, each with a tailored workflow that converges to the same analysis output.

HIP Workflow

1

Compile kernels with the instrumentation pass plugin: hipcc -fpass-plugin=AMDGCNSubmitAddressMessages.so -g ...

2

The compiler produces both original and instrumented (_inst) kernel variants in the same binary.

3

Run the application through omniprobe -i -a MemoryAnalysis ./app. The interceptor swaps to instrumented variants at dispatch time.

4

Analysis results are printed to console, grouped by source location.

Triton Workflow

1

No pre-compilation needed. Run the Triton script through omniprobe -i -a MemoryAnalysis -c /path/to/triton/cache script.py

2

Omniprobe sets LLVM_PASS_PLUGIN_PATH so Triton's JIT compiler loads the instrumentation plugin automatically during compilation.

3

The JIT-compiled kernels include both original and instrumented variants. The interceptor swaps at dispatch time, just like HIP.

4

Same analysis output as HIP. Zero manual instrumentation steps.

The Triton workflow is particularly powerful for AI/ML workloads. Developers can analyze Triton kernels generated from high-level Python code without any knowledge of LLVM IR, ISA, or manual compilation flags. Simply point omniprobe at the Triton script and cache directory.

Worked Example: Diagnosing Uncoalesced Memory Accesses

This example demonstrates Omniprobe's end-to-end workflow on a HIP kernel with a common memory access anti-pattern: column-major traversal of a row-major matrix.

Step 1 — The Problem Kernel

A kernel that reads a matrix column-by-column, causing each wavefront's lanes to stride across cache lines.

__global__ void column_sum(const float* matrix,
                          float* result,
                          int rows, int cols) {
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    if (col < cols) {
        float sum = 0.0f;
        for (int row = 0; row < rows; row++) {
            sum += matrix[row * cols + col];  // line 9: strided access
        }
        result[col] = sum;
    }
}

Adjacent threads (lanes) access addresses that are cols elements apart. On a 1024-column matrix with 128-byte cache lines, each wavefront of 64 lanes touches up to 64 separate cache lines instead of the minimum 2.

Step 2 — Compile and Run with Omniprobe

# Compile with instrumentation and debug info
hipcc -fpass-plugin=AMDGCNSubmitAddressMessages.so -g \
      column_sum.hip -o column_sum

# Run through Omniprobe
omniprobe -i -a MemoryAnalysis ./column_sum

Step 3 — Omniprobe Output

The memory analysis handler reports per-source-location efficiency metrics:

Source Location Access Type Cache Lines Touched Minimum Possible Ratio Verdict
column_sum.hip:9 GLOBAL load 64 2 32.0x Uncoalesced
column_sum.hip:11 GLOBAL store 2 2 1.0x Coalesced

Line 9 has a 32x memory bandwidth inefficiency. Line 11 (the store to result[]) is perfectly coalesced. The developer immediately knows where to focus optimization effort.

Step 4 — The Fix

Transpose the access pattern so adjacent lanes read adjacent elements:

__global__ void column_sum_fixed(const float* matrix,
                                float* result,
                                int rows, int cols) {
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    if (col < cols) {
        float sum = 0.0f;
        for (int row = 0; row < rows; row++) {
            sum += matrix[col * rows + row];  // transposed: coalesced
        }
        result[col] = sum;
    }
}

Re-running Omniprobe confirms the fix:

Source Location Access Type Cache Lines Touched Minimum Possible Ratio Verdict
column_sum.hip:9 GLOBAL load 2 2 1.0x Coalesced
column_sum.hip:11 GLOBAL store 2 2 1.0x Coalesced

Use Cases

HIP Kernel Optimization

A performance engineer suspects a HIP kernel is memory-bound but counter-based tools only show high L2 traffic. Omniprobe identifies the exact load instruction accessing scattered addresses, quantifies the cache-line waste, and attributes it to a source line. The fix is targeted and immediately verifiable.

Triton Kernel Tuning

A machine learning engineer writes a Triton attention kernel in Python. Omniprobe integrates with Triton's JIT compiler, instruments the generated GPU code automatically, and reports which memory access patterns in the generated code are inefficient — all without the developer needing to read LLVM IR or ISA.

Library-Level Analysis

A team needs to validate memory access efficiency across an entire GPU math library (e.g., rocBLAS, hipBLASLt). Omniprobe's library filter selects specific libraries for analysis while ignoring the rest. On-demand scanning keeps startup fast even with 12,000+ kernel symbols.

LDS Optimization

A kernel uses shared memory (LDS) for data reuse, but performance is worse than expected. Omniprobe's bank conflict analysis reveals that the access pattern causes 4-way bank conflicts on every shared memory read. The developer adjusts the padding to eliminate conflicts.

Getting Started

Prerequisites

ROCm 7.x
AMD CDNA GPU
CMake 3.21+
Triton 3.x (optional)

Build

# Clone with submodules
git clone --recursive https://github.com/your-org/omniprobe.git
cd omniprobe

# Configure
cmake -B build \
  -DROCM_PATH=/opt/rocm \
  -DCMAKE_HIP_ARCHITECTURES=gfx90a \
  -DCMAKE_INSTALL_PREFIX=$PWD/install

# Build and install
cmake --build build --parallel
cmake --install build

Quick Start: HIP

# Compile your kernel with instrumentation
hipcc -fpass-plugin=$(omniprobe --plugin-path AddressMessages) \
      -g my_kernel.hip -o my_app

# Analyze
omniprobe -i -a MemoryAnalysis ./my_app

Quick Start: Triton

# No pre-compilation needed
omniprobe -i -a MemoryAnalysis \
          -c ~/.triton/cache \
          python my_triton_script.py

Output Formats

Console (default)
Human-readable per-source-location report printed to stdout. Best for interactive debugging.
CSV (-f csv)
Machine-readable comma-separated output. Best for spreadsheet analysis and data pipelines.
JSON (-f json)
Structured JSON output. Best for automated tooling, CI integration, and programmatic post-processing.

Roadmap

Omniprobe v1.0 establishes the foundation for source-level GPU memory analysis. The following capabilities are planned for future releases:

1
v1.0 Release Stable release with full HIP and Triton support, memory analysis and basic block handlers, library filtering, and scope control. All APIs stabilized.
2
Agent-First JSON Output Structured JSON output for all handlers, designed for consumption by AI coding agents and automated optimization tools. Every handler emits machine-readable analysis results.
3
MCP Integration Layer Model Context Protocol (MCP) server enabling AI tools (Claude, Copilot, etc.) to invoke Omniprobe analysis directly. Run memory analysis from your IDE's AI assistant.
4
LDS Race Detection New analysis handler that detects data races in LDS (shared memory) by tracking read/write sets across wavefronts within a workgroup. Catches synchronization bugs that hardware counters cannot see.
5
Binary Instrumentation Instrument pre-compiled GPU binaries without access to source code or recompilation. Enables analysis of closed-source GPU libraries and vendor-provided kernels.
The MCP integration layer will make Omniprobe the first GPU memory analysis tool that can be invoked directly by AI coding assistants, enabling a new workflow where the AI agent identifies the performance problem, runs Omniprobe, interprets the results, and proposes a fix — all within a single conversation.
